End-to-end encrypted messaging system

I have an interest with computer security, as well as privacy. So, I decided to create this encrypted chat service to reflect on that , as well as gain experience in cryptography, web development, and web security.

The Threat Model

  • Server can not be trusted.
  • Nation state actors sniffing network.
  • DDOS attacks.
  • DNS spoofing.
  • Nation states actively blocking access.
  • Phishing/XSS attacks/CSRF attacks.
  • Client themselves are vunerable.

The design of the system was made with this threat model in mind.
Servers are always being exploited into, therefore, we must assume that the server is compromised at all times to ensure that users are kept safe. One of the most basic things that the server must do, is to keep all data encrypted on the server, with the users only having the ability to

Server cannot be trusted
In the event that the server is compromised, we want users to still be able to send and recieve messages while still having their identity anon. Additionally, messages sent should be confidential, and need to preserve intergity.
Therefore, encryption happens on the user's end. The client only ever sends encypted data to the server, on top of TLS. All data is then recieved by the end user, who verifies and decrypts the data.
Privilege Seperation
Services should be serperated based on the idea of least privilege.
If one service is exposed, others are not.
All services should talk via TLS. Example NSA Snowden leaks.
What about server sending things out of order? Prevent messages being sent
Idea: Order the messages, computationally impossible to try to reodrder messages.
Idea 2: Give clients/users a time stamp like TLS certificates. Where messages can time out.
Idea 3: Check for duplicates.
Ultimately, the server can deny service. Example: China and telegram
Can we do something about that? What about creating a peer to peer network? Use optional peer to peer?
